在真实的商业项目和一般的工作实践中，**仅靠你列举的那 3 张表（Credentials, Range, Schedule）搭建一个 PoC（概念验证）或极其简单的测试环境是足够的，但在企业级生产环境中是不足够的。**

在企业级生产环境中，事前配置阶段（第2阶段）为了保证**高可用性（防止单点故障）**和**扫描效率（防止盲目扫描导致网络拥堵）**，还**一定**会配置另外 2 张核心表。

我已经按照你要求的日语交付格式，为你补充了这两张在实际工作中必须要设定的表。你可以直接将它们追加到第2阶段的交付文档中：

---

データベーステーブル名: **`ecc_agent_cluster`**
業務上の名称: **MIDサーバクラスタ（MID Server Cluster）**
テーブルの役割と詳細説明:
**役割：** 複数のMIDサーバを論理的なグループにまとめ、負荷分散（Load Balance）やフェイルオーバー（Failover）を実現します。
**説明：** 一般的な本番環境では、単一障害点（SPOF）を防ぐため、`discovery_schedule`（実行計画）は単一のMIDサーバに直接割り当てるのではなく、必ずこの「クラスタ」に割り当てます。これにより、1台のMIDがダウンしても自動的に別のMIDがスキャンを引き継ぎます。
手動介入の要否:
**[手動設定]**
初期構築時やサーバー増減時に、管理者がクラスタを手動で作成し、対応するMIDサーバを所属させる必要があります。

データベーステーブル名: **`discovery_behavior`**
業務上の名称: **検出ビヘイビア（スキャン動作制御）**
テーブルの役割と詳細説明:
**役割：** 対象ネットワークに対するスキャンの「使用プロトコル」や「動作方針」を細かく制御します。
**説明：** デフォルトではすべてのプロトコル（SSH, WMI, SNMP等）を順番に試行しますが、これを使用すると「このセグメントはWindows専用なのでWMIのみ試行する」といった戦略を定義できます。これを`discovery_schedule`に紐づけることで、スキャン時間とネットワーク負荷を劇的に削減できます。
手動介入の要否:
**[手動設定]**
対象ネットワークの特性やセキュリティポリシー（FWのポート開通状況など）に応じて、管理者が最適な動作方針を手動で作成・適用します。

### 给你的内部解读（交付时无需向客户展示）：

* **为什么要加 `ecc_agent_cluster`：** 生产环境中绝对不允许 Schedule 直接绑在一台 MID Server 上。万一那台机器死机，当天的 CMDB 更新就全断了。客户如果在交付物里看不到 Cluster（集群）的配置，会认为你们的设计不够专业。
* **为什么要加 `discovery_behavior`：** 当企业扫描成千上万个 IP 时，如果每个 IP 都拿 SSH, WMI, SNMP 轮流撞一遍，扫描窗口期根本不够用，且会触发大量安全警报（IDS/IPS）。Behavior 是调优的必经之路。
