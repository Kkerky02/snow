
## 1. パターンベースのディスカバリとは？

**サービスマッピング（Service Mapping）の主要な手法**であり、アプリケーションサービスが依存している機器やアプリケーションのデータを自動的に収集する仕組みです。

* **パターン（Pattern）：** 一連の精密なコマンド（命令）のセットです。次の2つのコアアルゴリズムが含まれています。
* 構成アイテム（CI）の属性を識別する。
* CIの「接続先（送信接続）」（システム同士の呼び出し関係）を検出する。


* **最終目的：** データを収集した後、**CMDB（構成管理データベース）** 内に、視覚的な「アプリケーションサービスのトポロジマップ（構成図）」を自動作成して保存することです。

---

## 2. コアとなる仕組み（動作メカニズム）

* **探索の起点：** ディスカバリ（探索）のプロセスは、**「エントリポイント（Entry Point）」**（例：クライアントがサービスにアクセスする際のIPアドレスやホスト名）からスタートします。
* **2段階のディスカバリ：** 1. **横方向のディスカバリ（Horizontal Discovery）：** まず、サービスを稼働させているサーバーなどの「ホスト機器」をスキャンして特定します。
2. **トップダウンディスカバリ（Top-down Discovery）：** ホストが特定された後、その上で動いている具体的な「アプリケーション」や「システム間の接続関係」を深掘りしてマップ化します。
* **主要コンポーネント：** 全体のプロセスは、社内ネットワーク内に配置された **MIDサーバー（MID Server）** による実際の探索に大きく依存しています。また、ServiceNowクラウドとMIDサーバー間のメッセージ中継所として **ECCキュー（外部通信チャネルキュー）** を使用します。

---

## 3. 詳細なインタラクションとデータの流れ

全体のプロセス（11のステップとアーキテクチャ図）を簡潔にまとめると、以下の流れになります。

* **定義と確認：** 管理者がエントリポイントを指定すると、システムは該当するホストがすでにCMDBに存在するかを確認します。
* **ホストの探索：** 存在しない場合、ServiceNowは探索リクエスト（Shazzamプローブなど）をECCキューに送信します。MIDサーバーがそのタスクを受け取ってオープンポートをスキャンし、最終的にホスト情報をCMDBに書き込みます。
* **アプリと接続のマップ化：** ホストの準備が整うと、MIDサーバーは「パターン」の識別処理を実行してアプリのCIを検出します。その後、接続処理を実行して「接続先（送信接続）」を探します（この際、`cmdb_tcp` 表や `sa_flow_connection` 表のデータも参照し、トラフィック情報を補完します）。
* **マップ（トポロジ）の生成：** 検出されたすべてのCIと接続関係がCMDBに書き込まれ、最終的に完全な「アプリサービストポロジマップ」が自動的に描画されます。

Service Mapping が CMDB 内に該当アプリケーションのホストデバイスが存在しないと判断した場合、Discovery を起動して自動検出を行います。Discovery は ECC キューを介して MID サーバーに調査を指示し、対象ホストのポートスキャン、分類、識別を段階的に実行させます。収集された詳細情報は ECC キュー経由でシステムに戻されて CMDB へ自動登録され、最終的に Service Mapping によってホスト検出の完了が同期・確認されます。

CMDB 内でホストが特定されると、Service Mapping はそのホスト上で稼働するアプリケーション（CI）の検出プロセスへ移行します。ECC キューを介して MID サーバーに要求が割り当てられ、MID サーバーは特定のパターン（検出アルゴリズム）を実行してアプリケーション CI の属性を識別すると同時に、外部への送信接続（依存関係）を探索します。収集された情報は ECC キュー経由で戻され、Service Mapping が接続データを補完（cmdb_tcp などのテーブル情報を加味）した上で CMDB へ書き込み、サービスマップへと追加します。この探索は、接続先のアプリケーションに対しても依存関係がなくなるか境界（バウンダリ）に達するまで再帰的に繰り返され、トップダウンのサービスマップが完成します。

1. 水平検出 (Horizontal Discovery) ＝ 「IT資産の網羅的な棚卸し」
目的： 社内に存在するすべてのITインフラ（サーバー、ネットワーク機器、ストレージ等）を漏れなく洗い出すこと。

ロジック： 指定されたIPアドレス帯域（ネットワークセグメント）をスキャンし、そこに存在する（応答があった）すべての資産情報を機械的に収集します。

2. トップダウン検出 (Top-down Discovery / Service Mapping) ＝ 「業務サービスの関連性マッピング」
目的： 特定のビジネス（例：「モバイルバンキングサービス」など）を支えているすべての構成要素、およびそれらの相互の依存関係（データフロー）を特定すること。

ロジック： サービスの入り口（URL等のエントリーポイント）を起点として依存関係を順にたどり、その「業務に関わっている」資産のみをピンポイントで抽出・接続します。

Service Mapping（サービスマッピング / トップダウン検出）における最終的な成果を格納するコアテーブルです。このテーブルには、自動マッピングによって生成された「アプリケーションサービス（Application Service）」が保存されます。
【手動介入が必要な部分（管理およびビジネスとの整合）】


1. トポロジのレビューと承認（Approve）： サービスオーナー（Service Owner）は、自動生成されたサービスマップが実際の業務フローと一致しているかを手動で確認し、「承認」を行う必要があります。


2. CSDM上位層とのマッピング： アーキテクトまたは構成管理者は、この「アプリケーションサービス」を上位層の「ビジネスアプリケーション（Business Application）」に手動で紐付ける必要があります（例：「グローバルCRMシステム-本番環境」を「グローバルCRMシステム」という親カテゴリに関連付ける）。


3. 管理属性の補完： サービスオーナー、L1/L2サポートグループ、ビジネス上の重要度（クリティカリティ）など、システムがスキャンできない非技術的な管理フィールドを手動で入力・補完します。
